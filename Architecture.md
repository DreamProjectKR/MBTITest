# Architecture Evaluation

## 전제

이 평가는 **현재 저장소에 실제로 구현된 범위**를 기준으로 작성했습니다.

- 평가 대상: Cloudflare Pages + Worker + D1 + R2 + KV 기반의 MBTI 테스트 콘텐츠 관리/제공 시스템
- 비평가 관점: Architecture, SOLID Principle, Functional Programming Principle, Clean Code
- 제외: 구현되어 있지 않은 별도 사용자 계정/인증/권한 시스템

즉, 이 문서는 "콘텐츠 관리형 MBTI 플랫폼"으로서의 구조를 평가합니다.

## 객관적 총평

현재 프로젝트의 구조는 **작은 팀이 빠르게 운영할 수 있는 실용적인 edge-native 아키텍처**로는 꽤 괜찮습니다.
정적 자산은 Pages, API/에셋 프록시는 Worker, 메타데이터는 D1, 본문/이미지는 R2, 핫 리드는 KV로 분리되어 있어 인프라 역할이 비교적 명확합니다.

다만 코드베이스가 커질수록 다음 리스크가 분명합니다.

1. Worker 진입점과 admin 유틸 계층에 책임이 점점 모이기 쉽습니다.
2. D1/R2/KV를 함께 다루는 저장 흐름은 분산 트랜잭션이 없어서 일관성 관리가 어렵습니다.
3. 프런트엔드 admin 상태 관리가 전역 상태 + 수동 렌더링 중심이라 복잡도가 더 올라가면 유지보수 비용이 빠르게 증가할 수 있습니다.

결론적으로, **현재 규모에는 적절하지만 중간 규모 이상으로 확장되기 전 경계 재정리와 응용 서비스 계층 보강이 필요한 구조**입니다.

## 아키텍처 평가

| 항목          | 점수(10점) | 평가                                                                                                                                                    |
| ------------- | ---------: | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 시스템 분리도 |          8 | Pages/Worker/D1/R2/KV 역할 분리가 명확합니다. 배포 구조와 데이터 저장소 책임도 이해하기 쉽습니다.                                                       |
| 경계 명확성   |          7 | public API, admin API, asset proxy의 경계는 선명한 편입니다. 다만 admin 저장 흐름의 비즈니스 규칙이 핸들러와 유틸에 나뉘어 있어 도메인 경계는 약합니다. |
| 확장성        |          6 | 현재 기능 범위에서는 충분하지만, route 수와 저장 시나리오가 늘면 `worker/index.ts`와 `store.ts`가 병목 지점이 될 가능성이 큽니다.                       |
| 일관성/안정성 |          5 | D1/R2/KV를 모두 갱신하는 흐름에서 ACID 트랜잭션이 없으므로, 실패 복구 정책을 계속 수동으로 관리해야 합니다.                                             |
| 운영 적합성   |          8 | Cloudflare 친화적 구조, 캐시 계층 분리, `Server-Timing`/캐시 헤더 활용 등 운영 관점은 실용적입니다.                                                     |
| 문서 정합성   |          7 | 최근 문서와 실제 동작이 많이 맞춰졌지만, 일부 구조적 판단 기준은 아직 코드보다 문서가 먼저 끌고 가는 수준은 아닙니다.                                   |

## Architecture 관점 상세 평가

### 잘 된 점

1. **배포/런타임 구조가 명확합니다**
   Pages는 정적 파일, Worker는 API와 asset proxy, D1은 메타데이터, R2는 본문/이미지, KV는 상세 캐시로 책임이 선명합니다.

2. **읽기 경로 최적화 방향이 분명합니다**
   public 상세 조회가 KV → Cache API → D1/R2 순으로 흘러 성능과 비용 양쪽을 함께 고려한 구조입니다.

3. **에셋 전달 전략이 일관적입니다**
   `/assets/*` 프록시를 통해 same-origin 정책, 캐시 정책, ETag, Cache-Tag를 한 곳에서 다룹니다.

4. **admin/public 조회 경계가 최근 구조상 더 명확해졌습니다**
   public은 공개된 테스트만, admin은 draft 포함 전체를 다루는 식으로 API 의미가 정리되어 있습니다.

### 아쉬운 점

1. **애플리케이션 서비스 계층이 약합니다**
   저장/업로드 같은 중요한 유스케이스가 핸들러와 `store.ts`에 분산되어 있어, "한 기능의 전체 규칙"을 한 곳에서 읽기 어렵습니다.

2. **라우팅 진입점이 커질 가능성이 큽니다**
   `worker/index.ts`의 route table 패턴은 현재는 단순하고 좋지만, 엔드포인트가 늘면 중앙 집중형 dispatch 파일이 빠르게 비대해질 수 있습니다.

3. **분산 저장소 일관성이 구조적으로 어렵습니다**
   D1, R2, KV는 저장 특성이 다른데, 이를 조율하는 표준 transaction workflow 또는 command layer가 아직 얇습니다.

4. **프런트 admin은 수동 조합 비용이 큽니다**
   `main.js`, `forms.js`, `render.js`, `state.js` 분리는 좋지만, 상태 변경과 렌더 타이밍을 사람이 계속 맞춰야 하는 구조입니다.

## SOLID Principle 평가

| 원칙 | 점수(10점) | 평가                                                                                                                                                          |
| ---- | ---------: | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| SRP  |          7 | `router.ts`, `http.ts`, `assets/handler.ts`처럼 책임 분리가 잘 된 파일이 많습니다. 반면 `store.ts`와 일부 admin 핸들러는 책임이 넓어지는 조짐이 있습니다.     |
| OCP  |          6 | 라우트 추가는 비교적 쉽지만, 저장 규칙 변경은 기존 핸들러/유틸 수정 비중이 높습니다. 새로운 유스케이스를 "추가"보다 "수정"으로 처리하는 부분이 있습니다.      |
| LSP  |          8 | 구조적 타입과 단순한 함수 계약 중심이라 대체 가능성을 해치는 상속 문제는 거의 없습니다.                                                                       |
| ISP  |          7 | `_types.ts`에 인터페이스를 최소 단위로 유지하려는 방향은 좋습니다. 다만 admin 쪽 데이터 구조는 화면/저장/검증 관심사가 조금 섞입니다.                         |
| DIP  |          6 | Worker 환경 의존성을 `context.env`로 받는 점은 좋습니다. 하지만 고수준 유스케이스가 여전히 D1/R2/KV 세부 구현을 직접 아는 경우가 많아 완전한 역전은 아닙니다. |

### SOLID 총평

현재 프로젝트는 **"절차적이지만 분리된 함수들"** 위에 서 있습니다.
즉, 나쁜 구조는 아니지만 진짜 의미의 응용 서비스 계층과 정책 객체가 두껍지는 않습니다.
규모가 조금만 더 커지면 SRP와 DIP가 먼저 흔들릴 가능성이 큽니다.

## Functional Programming Principle 평가

| 항목             | 점수(10점) | 평가                                                                                                                                                          |
| ---------------- | ---------: | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 순수 함수 활용   |          8 | 경로 정규화, 헤더 생성, 라우팅 파싱, payload 변환 등 순수 함수가 분명히 존재합니다.                                                                           |
| 부작용 경계 관리 |          7 | R2/D1/KV 접근은 대체로 핸들러/스토어 함수에 모여 있습니다. 다만 일부 유스케이스는 read/validate/write/invalidate가 여러 함수로 흩어져 흐름 파악이 어렵습니다. |
| 불변성 유지      |          7 | Worker 쪽은 비교적 양호합니다. 프런트 admin도 spread를 많이 쓰지만 전역 상태 객체를 계속 덮어쓰는 패턴이라 강한 불변 모델은 아닙니다.                         |
| 함수 조합성      |          7 | 작은 helper 조합은 잘 되어 있습니다. 그러나 도메인 정책을 조합 가능한 함수 세트로 추상화한 수준까지는 아닙니다.                                               |
| 예측 가능성      |          6 | pure helper는 예측 가능하지만, 저장 시나리오처럼 외부 저장소가 여러 개인 흐름은 여전히 추론 비용이 큽니다.                                                    |

### FP 총평

함수형 원칙은 **부분적으로 잘 적용되어 있지만, 시스템 전체를 함수형으로 설계했다고 보기는 어렵습니다.**
특히 helper 수준에서는 좋고, 유스케이스 orchestration 수준에서는 아직 절차적 흐름의 비중이 높습니다.

## Clean Code 평가

| 항목             | 점수(10점) | 평가                                                                                                                  |
| ---------------- | ---------: | --------------------------------------------------------------------------------------------------------------------- |
| 이름 짓기        |          7 | `parsePath`, `withCacheHeaders`, `normalizeAssetKey`처럼 의도가 잘 드러나는 이름이 많습니다.                          |
| 파일 가독성      |          7 | 전반적으로 짧은 파일과 역할 분리가 유지됩니다. 다만 핵심 유스케이스 파일은 점차 길어지고 있습니다.                    |
| 중복 제어        |          6 | helper 추출이 어느 정도 되어 있지만, admin 쪽 입력/업로드/저장 흐름에는 반복되는 제어 패턴이 남아 있습니다.           |
| 에러 처리 명확성 |          7 | JSON 에러 응답 규칙이 비교적 일관적입니다. 다만 복수 저장소 갱신 실패 시 복구 정책은 아직 케이스별로 분산돼 있습니다. |
| 테스트 가능성    |          6 | 최근 테스트 스크립트가 추가되었지만 전체 구조가 처음부터 테스트 우선으로 설계된 느낌은 아닙니다.                      |

### Clean Code 총평

코드 스타일은 대체로 단정하고, helper 분리도 잘 되어 있습니다.
문제는 "더럽다"기보다 **중요한 비즈니스 흐름이 여러 파일로 퍼져 있어서 읽는 사람이 머릿속에서 재조합해야 한다는 점**입니다.

## 가장 좋은 점

**인프라 계층과 전송 계층의 역할 분리가 비교적 선명합니다.**

- Pages와 Worker의 역할이 겹치지 않습니다.
- Worker 내부에서도 router, http util, assets handler 같은 축이 비교적 또렷합니다.
- Cloudflare라는 특정 플랫폼을 활용하는 방식이 즉흥적이지 않고 꽤 실무적입니다.

## 가장 큰 구조적 리스크

**저장소가 여러 개인데 유스케이스 단위의 일관성 관리 계층이 아직 얇습니다.**

현재 구조에서 가장 위험한 지점은 "D1/R2/KV를 함께 갱신하는 흐름"입니다.
기능 수가 늘수록 각 핸들러가 조금씩 다른 순서와 복구 정책을 가지게 될 가능성이 큽니다.
이 문제는 버그가 생기면 코드 한 줄보다 운영 데이터에서 먼저 드러날 수 있습니다.

## 개선할 점

1. **유스케이스 중심의 application service 계층을 두는 것이 좋습니다**
   예: `saveTestDraft`, `publishTest`, `uploadResultImage`, `replaceQuestionImage` 같은 단위로 orchestration 함수를 두고, 핸들러는 요청/응답에만 집중하도록 줄이는 방식입니다.

2. **`store.ts`를 역할별로 분리하는 것이 좋습니다**
   현재 파일은 key 규칙, R2 read/write, D1 meta upsert, timestamp touch를 함께 다룹니다.
   `assetKeys`, `testRepository`, `imageRepository`, `testWriteWorkflow`처럼 분리하면 SRP가 더 좋아집니다.

3. **저장 일관성 규칙을 표준화해야 합니다**
   "선검증 → 본문 저장 → 메타 반영 → 캐시 무효화 → 실패 시 보상" 같은 공통 workflow를 문서와 코드 양쪽에서 고정해야 합니다.

4. **admin 상태 관리를 조금 더 명시적으로 바꾸는 것이 좋습니다**
   지금도 파일 분리는 되어 있지만, 상태 전이 규칙이 코드 구조로 강제되지는 않습니다.
   최소한 action 단위 업데이트 함수나 reducer 스타일을 도입하면 복잡도 증가를 늦출 수 있습니다.

5. **테스트 레이어를 기능 단위로 나누는 것이 좋습니다**
   현재는 중요한 회귀 테스트가 막 생기기 시작한 단계입니다.
   앞으로는 `public API`, `admin save`, `image upload`, `cache invalidation` 단위로 테스트 그룹을 분리하는 편이 낫습니다.

6. **문서가 설계 기준이 되도록 끌어올릴 필요가 있습니다**
   지금 문서는 설명서는 되지만, 아직 "이 구조를 벗어나면 안 된다"는 설계 제약 문서로까지는 충분히 강하지 않습니다.

## 최종 결론

이 프로젝트는 **작은 규모의 콘텐츠 관리형 서비스로서는 실용적이고 이해 가능한 구조**입니다.
Cloudflare 기반 아키텍처 선택도 일관되고, helper 분리와 캐시 전략도 생각보다 성숙합니다.

하지만 장기적으로 좋은 아키텍처가 되려면, 이제부터는 "파일 분리"보다 **유스케이스 경계와 일관성 정책을 먼저 설계하는 단계**로 넘어가야 합니다.
현재 상태를 한 문장으로 정리하면 다음과 같습니다.

**"기반 구조는 좋지만, 다음 성장 단계로 가기 위해서는 저장 흐름과 도메인 경계를 코드 구조로 더 강하게 표현해야 하는 프로젝트"** 입니다.
